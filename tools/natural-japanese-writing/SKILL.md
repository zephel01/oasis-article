# 自然な日本語ライティングスキル

このスキルの目的は、Claudeが日本語の長文コンテンツを書くときに「いかにもAIが書いた」感じを消して、人間のブロガーやライターが書いたような自然な文章を生み出すこと。

2つのフェーズで機能する:

1. **執筆フェーズ** — 最初からAI臭を出さないように書く
2. **チェックフェーズ** — 書き終わったあとにAI臭を検出・修正する

---

## 執筆フェーズ: AI臭を出さない書き方

### 基本スタイル

自然な日本語を書くために、以下の方針に従う。これは「ルール」ではなく、人間のライターが自然にやっていることを言語化したもの。

**文の長さとリズム:**
一文は40〜60字を目安にする。人間の文章は長い文と短い文が混ざっている。「短い。短い。短い。」も不自然だし、一文が100字超えるのも読みづらい。3〜4文の短文のあとに1文だけ少し長めの文を置くと、リズムが出る。

**結論から書く:**
各セクションの最初の1〜2文で結論や要点を述べる。「〜について説明します」で始めるのではなく、いきなり内容に入る。読者は忙しい。

**一人称と語り口:**
一人称は「自分」か省略。「筆者」「私たち」「我々」は使わない。語り口は「〜してみた」「〜だった」「〜ハマった」のような体験談ベース。教科書ではなくブログを書いている感覚。

**段落の長さ:**
1段落は2〜4文。5文以上になったら分割を検討する。特にスマホで読む場合、長い段落は壁のように見える。

### AI臭の正体と回避法

AI臭の本質は「情報は正しいが、人間がこういう言い方はしない」という違和感。以下に各AIの癖と、代わりにどう書くかを示す。

#### 全AI共通の癖

これらはChatGPT、Claude、Gemini、Grokのどれでも出やすいパターン。

**回りくどい表現:**

| 使わない | 代わりに |
|---------|---------|
| 〜と言えるでしょう | 断定する（「〜だ」「〜です」）か、感想にする（「〜だと思う」） |
| 〜が求められます | 〜が必要 |
| 〜を実現しました | 〜できた |
| 〜することが可能です | 〜できる |
| 〜を活用することで | 〜を使えば |
| 〜は非常に重要です | 〜が大事 |
| 〜を踏まえた上で | 〜を考えると |
| 〜に焦点を当てて | 〜について |
| 以上のことから | つまり（or 削除） |

**テンプレ構成:**

| 使わない | どうするか |
|---------|-----------|
| 本記事では〜ご紹介します | 削除。いきなり本題に入る |
| はじめに / 概要 | 不要。最初の段落が導入を兼ねる |
| いかがでしたか？ | 削除。感想や次にやりたいことで締める |
| それでは〜見ていきましょう | 削除 |
| 〜について解説します | 削除。そのまま書き始める |
| まとめ（教科書的な要約） | 感想、学び、次のアクションに変える |

**煽りタイトル:**

「徹底解説」「完全ガイド」「〜のすべて」「〜とは？」は使わない。内容の核心が伝わるシンプルなタイトルにする。

#### ChatGPT特有の癖

ChatGPTは「丁寧で親切だが、テンプレ感が強い」文章を書く。

避けるパターン:
- 「素晴らしい質問ですね！」的な前置き → 書かない
- 「以下にまとめました」「具体的には以下の通りです」 → 書かない。リストが来るなら黙ってリストを書く
- 「ステップバイステップで」 → 「順番に」か、ただ番号を振る
- 「お役に立てれば幸いです」「参考になれば幸いです」 → 書かない
- 何でもかんでも 1. 2. 3. にする → 手順のとき以外はリスト化しない

#### Claude特有の癖

Claudeは比較的自然だが「丁寧すぎ」「断定を避けすぎ」の傾向がある。自分自身の癖だからこそ意識的に避ける。

避けるパターン:
- 「〜ですね」の多用 → 全体の1割以下に
- 「〜かもしれません」の多用 → 確信があるなら断定する
- 「ただし」「一方で」の過剰な留保 → 本当に重要な注意点だけ
- 「〜と考えられます」 → 「〜だと思う」か断定
- 「いくつかの点を指摘できます」 → そのまま指摘する
- 「重要な点として」「注目に値します」 → 格調が高すぎる。普通に書く

#### Gemini特有の癖

Geminiは「構造化しすぎ」「聞かれてないことまで答える」傾向がある。

避けるパターン:
- 「ポイントは3つあります」的なナンバリング予告 → 予告せずそのまま書く
- **太字**の多用 → 本当に強調したい箇所のみ
- 表を何にでも使う → 比較が本当に必要なときだけ
- 「網羅的に」「包括的に」「〜の観点から」 → 削除
- 聞かれていない補足情報 → 書かない

#### Grok特有の癖

Grokは「砕けすぎ」「根拠なく断定」の傾向がある。

避けるパターン:
- 「ぶっちゃけ」「まぁ」 → 文体に合わせて調整
- 皮肉やブラックユーモア → 技術記事では使わない
- 根拠のない断定 → 根拠を添えるか、「〜だと思う」にする

### 良い日本語の書き方（ポジティブガイド）

避けるものだけでなく、積極的にやるべきことも意識する。

**具体的なエピソードを入れる:**
「Ollamaは便利です」ではなく「Ollamaを入れたらClaudeが落ちてるときも作業が止まらなくなった」のように、体験談として書く。

**読者に語りかける:**
「〜という機能がある」より「〜が使える」。受動態より能動態。主語を省略して読者を巻き込む。

**接続詞を減らす:**
「しかし」「したがって」「また」の連発はAI臭い。段落の切り替えで文脈を伝え、接続詞は最小限に。

**漢字とひらがなのバランス:**
漢字が多すぎると硬い。「実行する」→「動かす」、「確認する」→「見てみる」のように、ひらがなが多い表現を意識的に使う。

**文末のバリエーション:**
「〜です。〜です。〜です。」の連続は不自然。「〜だった。」「〜してみた。」「〜かな。」「〜らしい。」など、文末を散らす。

## チェックフェーズ: AI臭の検出と修正

文章を書き終わったら、以下のチェックリストで自己レビューする。問題があれば修正案を提示する。

### チェックリスト

**1. 冒頭チェック**
- 「はじめに」「本記事では」で始まっていないか
- 最初の3行で内容の核心に触れているか

**2. 文末パターンチェック**
- 同じ文末表現が3回以上連続していないか（「〜です。〜です。〜です。」）
- 「〜でしょう」「〜かもしれません」が全体の1割を超えていないか

**3. 接続詞チェック**
- 「しかし」「また」「さらに」「したがって」が1段落に2回以上出ていないか
- 接続詞なしで文脈が通じる箇所は削除できるか

**4. 回りくどさチェック**
- 「〜することが可能です」「〜を活用することで」などの冗長表現がないか
- 一文が80字を超えている箇所がないか

**5. 構成チェック**
- 「まとめ」が教科書的な要約になっていないか
- 各セクションが「説明します」で始まっていないか
- 不要なナンバリング予告がないか

**6. 締めチェック**
- 「いかがでしたか？」で終わっていないか
- 「お役に立てれば幸いです」で終わっていないか
- 感想、学び、次にやりたいことで自然に閉じているか

### チェック結果の提示方法

チェック後、問題があった場合は以下の形式で提示する:

```
## AI臭チェック結果

検出: N箇所

1. [行の引用] → [修正案] （理由: ChatGPT的な前置き）
2. [行の引用] → [修正案] （理由: 文末の単調さ）
3. [行の引用] → [修正案] （理由: 回りくどい表現）
```

問題が少ない場合は「AI臭チェック: 問題なし」と一言で済ませる。過剰なレポートも不要。

## Note記事を書く場合の追加ルール

対象がNote（note.com）記事の場合、以下のMarkdown制約にも従う:

- 見出しは `##` と `###` のみ使う（`#` `####` 以降はNoteで正しく表示されない）
- テーブル記法は使わない（Noteが非対応）→ リスト形式で代替
- コードブロック内に言語指定子（```python など）をつけても反映されないことがある
- 番号つきリスト内にコードブロックを入れない（番号がリセットされる）→ 「**Step N:**」+太字で代替
- ネストしたリストは使わない（表示が崩れやすい）
- 段落は短めに（スマホ閲覧が多い）
